# 会话优化

## 0. 当前实现状态

当前 ACP 会话链路已经能工作，并已从轮询刷新切到事件级 push；后续重点是继续压实 timeline / snapshot 作为会话主路径，降低历史读取和检索成本。

当前 Gold Band 现状：

- 会话主数据文件为：`acp.session.json`、`acp.events.jsonl`、`acp.raw.jsonl`、`acp.diagnostics.jsonl`。
- `acp.raw.jsonl` 记录完整 ACP inbound / outbound raw frame，见 `src/acp/events.rs:48-52`、`src/acp/client.rs:303`、`src/acp/client.rs:931`。
- `acp.events.jsonl` 当前仍以逐帧归一化 event 为主，`normalize_session_update` 会把整份 `update.clone()` 塞进 `event.raw`，见 `src/acp/events.rs:165-199`。
- 会话详情读取时，后端会再次扫描 `acp.events.jsonl`，把连续 text / thought delta merge 成窗口，再计算 usage、permission 和 session elapsed，见 `src-tauri/src/view_models.rs:3160-3329`、`src-tauri/src/view_models.rs:3595-3616`。
- 前端打开 ACP 会话后常驻订阅 Tauri `gold-band://acp-session-updated`；活跃会话每个后端聚合 timeline item 以单事件 payload 推送到当前窗口，完整 session 快照只用于首次打开、历史查询和命令完成后的最终同步，不再用轮询或每帧重建完整 session 批量刷新。
- 会话式运行页在 run 已 running 但首个 attempt 尚未出现在 session tree 前展示 `Agent 调起中`；attempt 已创建但尚无可见 timeline item 时展示 `处理中...`，首个 thought 到达后切换为 `思考中...`。运行状态标签使用轻量 CSS 圆环旋转，避免 SVG spinner 在高频状态刷新时卡顿。
- 历史分页请求已切到 `beforeCursor / afterCursor` 主路径，同时兼容旧 `beforeSeq / afterSeq`；前端聊天列表使用原生滚动容器触发顶部/底部分页，prepend 旧页前捕获当前可见 timeline item 的 DOM 锚点，合并后按同一锚点的新旧 top 差值保位。
- session token / cost 既在 runtime 内存里跟踪，也在读取 `acp.events.jsonl` 时重算一遍，见 `src/acp/client.rs:859-880`、`src-tauri/src/view_models.rs:3215-3231`。
- session 净处理耗时依赖 `AcpSessionElapsedState` 在读 events 时重建，包含 permission wait 扣减逻辑，见 `src-tauri/src/view_models.rs:3435-3528`。

当前方案的主要问题：

1. `acp.events.jsonl` 与 `acp.raw.jsonl` 重复偏重，`events` 没有真正成为“UI 已加工时间线”。
2. 活跃会话已改为事件级 push，但 session 状态、usage 和检索索引仍需要继续压实权威来源。
3. 读取详情、分页、统计仍有部分逻辑依赖重新扫描和 merge，history 链路仍需要继续从 legacy event 迁到 timeline / snapshot 主路径。
4. ACP stop 的运行态与 provider 退出时机必须解耦：UI / runtime 先立即收敛到 `paused + process_interrupted`，provider 再走 `session/cancel` 优雅退出，超时后 kill fallback，避免会话框长时间停留在 stopping。
5. token / elapsed 的权威来源不清晰，恢复与展示成本高。
6. 后续跨 session 搜索如果继续纯扫文件，复杂度和性能都会快速恶化。

## 1. 核心方向

本次会话优化的核心目标：

1. **活跃会话从 polling 改为 push。**
2. **会话时间线从逐帧 event 改为聚合后的 timeline item。**
3. **token / elapsed 从读时重算改为 runtime 聚合 + snapshot 持久化。**
4. **文件仍然是会话事实源，SQLite 只用于辅助检索。**
5. **旧会话可继续打开，通过 legacy reader / lazy migration 兼容。**

目标分层：

```text
ACP raw frame
  -> live session aggregator
  -> timeline final item
  -> push to UI
  -> rewrite timeline / update snapshot
  -> optional SQLite search index
```

这里必须明确三个边界：

- `raw`：协议排障事实源。
- `timeline`：UI 聚合时间线事实源。
- `SQLite`：辅助检索索引，不承担会话恢复和详情展示的唯一事实源。

## 2. 设计原则

### 2.1 会话详情优先服务 live UX

会话主路径必须优先服务：

- 文本流顺滑增长
- tool call 原地刷新
- permission 状态即时切换
- 历史向上翻页不抖动
- 会话恢复后可继续增长

因此主详情链路不能继续依赖“轮询文件 + 再 merge”。

### 2.2 timeline 必须是 UI 已加工结果

`acp.timeline.jsonl` 不再保存 raw-ish event 镜像，而应保存：

- 一条用户消息
- 一条 assistant 文本块
- 一条 thought 块
- 一条 tool call
- 一条 completed plan 块
- 一条 permission 块
- 必要的 system / error / compaction 块

即：一个逻辑 UI item 在内存中可以被多次更新，但落盘后只保留当前最终态，不能退回成 ACP chunk 日志。

### 2.3 文件优先，数据库旁路

当前项目 runtime 已按 attempt 目录组织，会话与 task / run / round / node / attempt 绑定紧密；因此本期不引入“DB 作为会话主存储”。

本期约束：

- `raw / timeline / snapshot` 为主数据。
- SQLite 仅承担跨 session prompt / timeline 检索索引。
- 没有 SQLite 时，详情、恢复、排障都必须继续工作。

### 2.4 破坏式升级，但保留旧会话可读

开发阶段允许替换旧方案，但不能让已有会话不可打开。

因此策略为：

- 新会话走 V2。
- 旧会话走 legacy reader。
- 必要时按首次打开或后台任务做 lazy migration / backfill。

## 3. 目标文件结构

每个 ACP attempt 的目标文件结构：

```text
attempt/
  worker-ref.json
  acp.raw.jsonl
  acp.timeline.jsonl
  acp.snapshot.json
  acp.diagnostics.jsonl
  provider.pid
```

说明：

- `worker-ref.json`：继续承载 session id、continue 信息、provider handoff 信息，是 session identity 的事实源。
- `acp.raw.jsonl`：完整 ACP inbound / outbound raw frame，只用于协议排障与极端恢复。
- `acp.timeline.jsonl`：UI 聚合时间线的 final item 列表，每个逻辑 item 一行，仅保留当前最终态。
- `acp.snapshot.json`：当前会话聚合快照、metrics、游标与恢复锚点；用于取代现有 `acp.session.json`。
- `acp.diagnostics.jsonl`：adapter / protocol diagnostics。

本期不再把 `acp.events.jsonl` 作为 V2 会话主路径。旧会话继续支持读取 `acp.events.jsonl`。

## 4. 目标架构

### 4.1 live session aggregator

后端新增 ACP 会话聚合器（以下简称 aggregator），负责：

1. 接收 ACP raw session updates。
2. 将 update 聚合进内存中的 session state。
3. 生成稳定的 timeline final item。
4. push 给前端。
5. 同步重写 `acp.timeline.jsonl`。
6. 同步更新 `acp.snapshot.json`。
7. 异步更新 SQLite 检索索引。

aggregator 是活跃会话的唯一 live 聚合层。前端不再直接解释 ACP 原始 frame，也不再依赖扫描 timeline 来获得最新 UI 状态。

### 4.2 前端改为 push 消费

活跃会话改为：

- 前端打开会话时订阅该 attempt 的 live channel，订阅生命周期跟随抽屉会话而不是发送按钮锁定状态，避免漏掉调起后的早期帧。
- 后端对每个聚合后的 timeline item 直接推送单事件 payload：新增 item、更新 item、权限请求和工具/文本流状态变化；手动 `send_acp_prompt` 与自动工作流 provider 执行路径都必须接入同一个 live event sink，不能只在命令完成后靠完整 session 快照刷新。
- 前端只 merge 对应 event；命令返回、节点完成或重新打开时再用完整 session 快照做最终同步，不在每个 ACP frame 上全量重拉；如果 live event 早于完整 session 快照到达，前端用最小 live session shell 承载当前窗口事件，避免首个流式片段因 `session` 为空而不可见；抽屉关闭再打开同一 attempt 时先恢复本地事件窗口并建立订阅，再主动补拉最新 session window，避免等待快照期间空白或旧快照覆盖 live tail。

历史加载和首次打开仍然走命令查询，但活跃期间的新增变化必须由 push 主导。ACP Chat 主列表渲染以聚合后的 timeline item 为最小视觉单元，窗口内使用浏览器原生滚动；不把一条 assistant/user 消息拆成多个可见 row，也不把 thought/tool 的 summary/detail 拆成两条消息。窗口裁剪和历史分页只影响已加载 item 集合，不能破坏单条消息或单张工具/思考卡片的视觉连续性；运行中的 thought、tool call 和子 Agent 分组默认保持收起，只更新紧凑标题与状态，用户可主动展开查看详情。

### 4.3 history 读取改为 snapshot + timeline

打开会话详情时：

1. 先读 `acp.snapshot.json`，获得摘要、metrics、active 状态、游标。
2. 再从 `acp.timeline.jsonl` 读取最近一页已聚合 item。
3. 如果会话仍活跃，再建立 live subscription。

历史翻页时：

- 基于 timeline cursor / checkpoint 向前或向后读取。
- 不再对 `acp.raw.jsonl` 或逐帧 event 做窗口 merge。

## 5. `acp.timeline.jsonl` 设计

### 5.1 设计目标

`acp.timeline.jsonl` 必须满足：

- 支持一行一个最终逻辑 item。
- 支持同一逻辑 item 在内存中多次更新，但文件中只保留当前最终态。
- 支持会话恢复时快速重建最终时间线。
- 支持历史分页。
- 支持与 live item model 保持一致。

### 5.2 item 粒度

timeline item 粒度固定如下：

- `userMessage`
- `assistantMessage`
- `assistantThought`
- `toolCall`
- `plan`
- `permissionRequest`
- `systemNotice`
- `sessionError`
- `contextCompaction`

### 5.3 final item 模型

`acp.timeline.jsonl` 推荐每行一个最终 item，示意：

```json
{
  "item": {
    "id": "assistant-msg-12",
    "kind": "assistantMessage",
    "status": "streaming",
    "content": "我先检查当前工作流节点配置...",
    "startedSeq": 120,
    "endedSeq": 127,
    "startedAt": "1717612345Z",
    "endedAt": "1717612348Z",
    "meta": {
      "attemptId": "..."
    }
  }
}
```

约束：

- `id` 是稳定标识，同一逻辑消息 / thought / tool call 在整个 turn 中保持不变。
- `startedSeq` 表示逻辑 item 首次出现时关联的 source seq。
- `endedSeq` 表示该 item 当前最终覆盖到的最新 source seq。
- `startedAt / endedAt` 用于展示、耗时、恢复和分组。
- 旧 patch log 仅作为 legacy reader 兼容输入，新写入统一采用 final item 结构。

### 5.4 文本 / thought 聚合策略

文本与 thought 必须在写时聚合，而不是读取时 merge：

- 第一帧 `AgentMessageChunk` 到来时，新建一个 `assistantMessage` item。
- 后续属于同一稳定流的 chunk 到来时，更新该 item 的 content / endedSeq / endedAt。
- `AgentThoughtChunk` 同理写入 `assistantThought` item。
- `usageUpdate`、`modeUpdate`、`availableCommandsUpdate` 这类隐藏状态帧不能打断同一文本 / thought 流。
- 只有跨 turn、跨类型、跨 stable stream 时才新开 item。

### 5.5 tool call 聚合策略

tool call 必须按 `toolCallId` 原地聚合：

- `ToolCall` 创建时建立 `toolCall` item。
- `ToolCallUpdate` 到来时按 `toolCallId` 更新同一 item。
- terminal / diff / locations / raw input / raw output 等详情作为该 item 的字段或子结构更新。
- 不能把 `toolCall` 和 `toolCallUpdate` 分裂成多张卡。

### 5.6 permission / plan 聚合策略

- `permissionRequest`：pending -> selected / rejected 更新同一 item。
- `plan`：允许增量更新，但最终 UI 投影仍然是单个 `plan` item 或单个 completed plan block。
- `systemNotice` / `sessionError`：一条逻辑提示一个 item，不复用 raw error 行。

### 5.7 前端 Plan/Todo 面板（2026-06-08）

- 主会话顶层 plan 事件不再渲染为消息流中的独立卡片，而是聚合为 composer 上方单一可折叠面板：
  - 默认收起，展开后展示条目列表与状态 Badge。
  - 收起状态显示完成数/总数及进行中任务摘要。
  - 每次 plan 更新实时刷新面板内容，不再在消息流中重复追加块。
- 子 Agent 内部 plan 保留在对应分组内，仅主会话顶层 plan 被提升。
- 面板每次 render 取最后一个顶层 plan 事件的 entries 作为当前状态。
- 顶层 plan 事件仍写入 timeline 文件（保留完整性），但在前端消息流中被过滤。

### 5.8 前端工具调用参数展示（2026-06-08）

- 工具调用卡片展开后以有序参数列表展示输入：
  - 参数来源优先级：rawInput > title 解析 > locations fallback。
  - 同标签参数保留多个不同值，不再用 `Object.fromEntries` 去重。
  - 无结构化参数但有原始 input payload 时回退展示 JSON。
  - 扩展 `toolDetails` 提取来源：增加 `toolCall.input`、`toolCall.fields` 回溯。

`acp.snapshot.json` 是 V2 会话快照文件，用于取代现有 `acp.session.json`。

它不再只是 adapter/status metadata，而是统一承载：

- 会话摘要
- 聚合 metrics
- timeline 恢复锚点

### 6.3 读取优先级

1. `acp.session.json`
2. `acp.events.jsonl`
3. `worker-ref.json`
4. `acp.raw.jsonl`

## 7. live push 设计

### 7.1 推送内容

后端向前端推送的消息类型：

- `timelinePatch`
- `sessionStatusPatch`
- `usagePatch`
- `metricsPatch`
- `permissionPatch`
- `sessionErrorPatch`

推荐统一成：

```json
{
  "attemptKey": "...",
  "patchType": "timelinePatch",
  "payload": { ... }
}
```

### 7.2 推送节流

文本流不能按 token 级频率直接刷前端。建议：

- 文本 / thought patch：50~150ms 内合并后推一次。
- tool call / permission / status / error：即时推送。
- snapshot 写盘频率可略低于 UI push，但不得太慢，避免崩溃丢失恢复锚点。

### 7.3 断线恢复

前端重新订阅后：

1. 先取最新 `snapshot + timeline window`。
2. 再接入 live push。
3. 如果服务端保存了最近 patch revision，可在必要时从某 revision 之后补拉。

## 8. 历史分页设计

### 8.1 不再使用 `beforeSeq / afterSeq`

V2 不再把 raw frame seq 作为历史分页主游标。

原因：

- seq 是协议帧粒度，不是 UI item 粒度。
- 一个 UI item 可能覆盖多个 raw seq。
- 活跃会话中 seq 会持续增长，作为 UI 分页 anchor 不稳定。

### 8.2 使用 timeline cursor

历史分页改为：

- `beforeCursor`
- `afterCursor`
- `pageSize`

cursor 推荐编码：

- item ordinal
- byte offset checkpoint
- revision / updatedAt

实现上允许 cursor 是不透明字符串，不向前端暴露内部结构。

### 8.3 checkpoint / index

文件方案下，为避免从头扫描 timeline：

- 每 N 个 timeline item 生成一个 checkpoint。
- checkpoint 可内嵌在 `acp.snapshot.json` 尾部摘要，也可独立为 `acp.timeline.index.json`。

index 最小信息：

- checkpoint item ordinal
- file byte offset
- timestamp
- itemId

这样打开最新页、向上翻历史都不需要全量扫描。

### 8.4 无感丝滑分页

前端分页交互要求：

- 前端会话历史分页的单次加载条数由 `configs/app-config.json` 的 `acpChatEventPageSize` 控制，单位是聚合后的 timeline item；前端额外保留有限多页缓冲，避免小页大小下滚动跳变。
- 向上翻页只 prepend timeline item，不触发底部自动滚动。
- 保持 scroll anchor，避免闪跳。
- 活跃会话若用户在底部，live patch 到来后保持吸底；若用户在读历史，则只提示有新消息。
- 活跃会话的 live patch 与历史分页必须可以无缝拼接，因为它们使用相同的 timeline item 模型。

## 9. token / cost / elapsed 设计

### 9.1 token / cost

切到 V2 后，不再依赖扫 timeline 全量重算 session usage。

权威来源改为 runtime aggregator + snapshot：

- `usedTokens`
- `accumulatedUsedTokens`
- `contextWindowSize`
- `inputTokens`
- `outputTokens`
- `cachedReadTokens`
- `cachedWriteTokens`
- `totalTokens`
- `totalCostUsd`

规则：

- `usage_update` 实时更新 `usedTokens / contextWindowSize / totalCostUsd`。
- prompt response 结束时补齐 `inputTokens / outputTokens / cached* / totalTokens`。
- `accumulatedUsedTokens` 继续保留“只累计正向 delta”的策略，避免 compaction reset 冲掉 session 总用量。
- UI 直接读 snapshot 中的 usage，live 期间通过 `usagePatch` 刷新。

timeline 只在需要消息级 token 展示时，给最终 assistant message item 写入：

- `usedTokensAtStart`
- `usedTokensAtEnd`
- `tokenDelta`

### 9.2 elapsed

V2 下 session 净处理耗时必须改为 runtime 维护，不再以“详情页读文件重算”为主路径。

建议 runtime 维护：

- `sessionElapsedMs`
- `currentTurnElapsedMs`
- `activeTurnStartedAt`
- `activeTurnLastActivityAt`
- `permissionWaitStartedAt`
- `permissionWaitAccumulatedMs`
- `pendingPermissionIds`

规则保持与当前口径一致：

1. Gold Band 用户 prompt accepted 时开启新 turn。
2. assistant / thought / tool / plan 的有效活动刷新 `lastActivityAt`。
3. permission pending -> selected 期间不计入净耗时。
4. session active 时 `currentTurnElapsedMs` 可按 now 实时增长。
5. turn 结束时把净耗时累加进 `sessionElapsedMs`。

历史读取时：

- 优先读 snapshot 中的 metrics。
- 仅在 legacy 会话、snapshot 缺失或调试校验时，从 timeline / events 回放重建。

## 10. SQLite 辅助检索方案

### 10.1 原则

SQLite 在本方案中**只用于辅助检索**，不承担：

- 会话详情渲染主存储
- 活跃会话 live state
- raw frame 排障
- timeline 恢复的唯一来源

文件仍然是权威事实源。

### 10.2 使用场景

SQLite 只承担两类检索：

1. **跨 session 用户 prompt 搜索**
2. **可选的 final timeline item 搜索**

### 10.3 最小表设计

建议最小化为：

- `sessions`
  - `session_id`
  - `attempt_path`
  - `task_id / run_id / round_id / node_id / attempt_id`
  - `created_at`
  - `updated_at`
  - `title`

- `session_prompts`
  - `id`
  - `session_id`
  - `prompt_id`
  - `timestamp`
  - `text`
  - `normalized_text`

- `session_prompts_fts`（FTS5）
  - 对 `text` 建立全文索引

可选：

- `timeline_items`
  - 只存 final timeline item
  - 不存中间流式 revision

### 10.4 写入策略

- 文件写入成功后，再异步写 SQLite 索引。
- DB 写失败不影响会话详情与恢复。
- 可通过后台 backfill 为旧会话补索引。

### 10.5 兼容策略

- 新 V2 会话：写文件 + 写 SQLite 索引。
- 旧 V1 会话：首次命中搜索范围时可 lazy backfill。
- 搜索优先走 SQLite，详情优先走文件。

## 11. 旧会话兼容与迁移

### 11.1 版本定义

- **V1**：`acp.session.json` + `acp.events.jsonl` + `acp.raw.jsonl`
- **V2**：`acp.snapshot.json` + `acp.timeline.jsonl` + `acp.raw.jsonl`

### 11.2 兼容读取

读取逻辑：

1. 若存在 `acp.snapshot.json`，按 V2 打开。
2. 否则若存在 `acp.events.jsonl`，按 V1 legacy reader 打开。
3. 若用户显式打开旧会话，可在后台生成 V2 timeline / snapshot。

### 11.3 lazy migration

推荐 lazy migration，而不是强制一次性全量迁移：

- 第一次打开旧会话时：
  - 读取 V1 `events + session + worker-ref`
  - 生成 V2 `timeline + snapshot`
  - 后续优先走 V2

- 正在运行中的旧会话不强切：
  - 已启动的 V1 attempt 继续按旧链路跑完
  - 新建会话统一走 V2

## 14. 风险与权衡

### 14.1 风险

1. live push + history page 混合后，前后端状态同步复杂度上升。
2. timeline item 边界若定义不好，可能出现消息合并错误或 tool call 错位。
3. snapshot 与 timeline 若一致性处理不好，恢复链路会出现偏差。
4. SQLite FTS 对中文 prompt 的检索效果需要单独评估 tokenizer 与标准化策略。

### 14.2 权衡

- 本方案刻意不把 SQLite 升格为主存储，换来较低改动和更稳兼容，但历史查询高级能力会受限于“旁路索引”模式。
- 文件优先意味着仍需做好 checkpoint / index，否则 timeline 超大后历史读取仍会变慢。
- 相比当前方案，本方案实现复杂度上升，但会显著改善流式体验、职责边界与后续搜索扩展性。

## 15. 最终结论

本期推荐的会话优化方案是：

1. **以 `acp.timeline.jsonl` 取代 `acp.events.jsonl` 作为 V2 会话主时间线。**
2. **以 `acp.snapshot.json` 取代 `acp.session.json` 作为 V2 会话快照与恢复锚点。**
3. **保留 `acp.raw.jsonl` 作为协议排障事实源。**
4. **活跃会话从轮询改为 push，timeline item 采用写时聚合。**
5. **token / elapsed 改为 runtime 聚合 + snapshot 持久化，不再以读时重算为主路径。**
6. **SQLite 仅用于辅助检索，尤其是跨 session 用户 prompt 搜索，不承担会话主存储。**
7. **旧会话通过 legacy reader 与 lazy migration 继续可打开。**

## 16. 当前实现进度

已完成第一阶段落地：

- runtime attempt 路径已新增 `acp.snapshot.json` 与 `acp.timeline.jsonl`。
- ACP runtime 新建会话已切到仅写 `acp.snapshot.json` + `acp.timeline.jsonl`；不再为新会话创建 `acp.session.json` / `acp.events.jsonl`。
- ACP 会话读取已改为优先走 `snapshot + timeline`，仅旧会话缺失 V2 文件时回退 `session + events`。
- 前端 ACP Chat 活跃会话已切到 Tauri `gold-band://acp-session-updated` push 主路径；轮询兜底已移除，抽屉打开期间常驻订阅，重开会话仍通过 `snapshot + timeline` 初始化，再无缝接入 live push。
- ACP runtime 每次持久化聚合后的 timeline item 后都会推送单事件 payload；手动 ACP prompt 和自动工作流节点执行都接入同一个 live event sink，前端按稳定 event key merge 到当前窗口，命令或节点完成后的完整 session 快照只做最终同步，不再在每个 ACP frame 上重建整段 session。
- 会话式主页创建/重跑 run 也必须使用同一个 live event sink，并以后台 run 启动返回初始 VM；前端收到当前 run 的首个 attempt live event 后刷新 session tree 并选中该 session，避免新 UI 只等定时刷新或权限请求时才一次性补齐历史。
- 会话式运行页使用 compact composer 时，`AcpUsagePanel` 不得只展示 token 和计时；运行中必须在同一区域展示具体处理状态标签，并优先使用当前步骤语义：发送中、处理中、思考中、工具调用中、响应中、停止中。该状态仍属于 composer，不进入消息流。
- 历史分页请求已支持 `beforeCursor / afterCursor`，同时兼容旧 `beforeSeq / afterSeq`；前端 cursor 从当前已加载窗口的首尾事件计算，避免使用过期 session page cursor 重复拉错段。
- ACP Chat 历史区已改为原生滚动容器；向上接近顶部触发 `beforeCursor` 历史加载，向下到底部且存在较新窗口时触发 `afterCursor` 加载。
- 针对 prepend 历史页和窗口裁剪导致阅读位置漂移的问题，前端在加载前捕获当前可见 timeline item 的 DOM 锚点，合并旧页后按该锚点新旧 top 差值补偿 `scrollTop`；窗口内直接渲染聚合后的 timeline item，不再使用 `react-virtuoso`、render block 分片、预渲染像素或消息拆段补丁，assistant/user 消息、thought、tool call 和子 Agent 分组都保持单个视觉单元。
