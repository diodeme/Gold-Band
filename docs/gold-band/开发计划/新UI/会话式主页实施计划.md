# Gold-Band 新会话式 UI 完整方案

## Context

当前 Gold-Band 的主界面是工作台式任务编排，入口集中在 `web/src/pages/TaskListPage.tsx`、`web/src/pages/WorkflowPage.tsx`、`web/src/pages/RoundDetailPage.tsx`，用户需要理解 task / workflow / run / round / node / attempt 后才能进入会话。新方案的目标是新增一个以 chat bot 为主心智的会话式 UI，让用户从“输入需求 → 校验配置 → 创建 task + run → 进入当前 session 对话”开始，同时保留现有工作台 UI 并通过显式 toggle 切换。

本方案按用户澄清后的决策执行：默认进入会话模式、toggle 选择持久化、旧 task/run 仍显示但完整体验以新会话为主、左侧是会话列表、右侧主区域永远展示当前具体 session 对话，round/node/attempt 只作为 session 切换层级。

## Product Shape

### App shell

- 新增两种桌面形态：`conversation` 与 `workbench`。
- 首次启动默认进入 `conversation`；用户通过 Workbench / Conversation toggle 切换后立即持久化。
- Toggle 优先：最终展示会话形态还是工作台形态由持久化 toggle 决定，内部路径只作为当前页面对象的定位信息；根路径、刷新和重启都恢复上次 toggle 形态。
- 工作台现有页面保留，不改旧交互心智；会话式产品文档新增独立文档，不在旧文档上改写。

### Conversation sidebar

左侧栏结构：

1. 顶部：品牌区 + Workbench/Conversation toggle。
2. 快捷入口：快速对话、搜索、Agent 管理、上下文管理、运行模式管理。
3. 置顶区：显示用户置顶的会话，支持手动拖拽排序；置顶会话仍在原工作空间下重复显示。
4. 工作空间/项目区：支持多个 workspace 并行，每个 workspace 下列出 task/conversation。

会话行只显示三类信息：

- 标题
- 状态小圆点
- 相对时间

状态与排序规则：

- 最近排序只看最新 run 的发起时间，不看消息到达时间。
- 运行中不显示时间；已结束才显示分/时/天/周/月/年。
- 状态使用最新 run 的最终状态，复用现有 run status/outcome：成功为绿，失败/异常/停止为红，暂停为黄。

### Conversation home

- 新会话输入框支持任意需求文本。
- 新会话创建前可选择 workspace、运行模式 AUTO / WORKFLOW。
- AUTO 模式下显示 agent 与模型选择；agent 来自 Agent 管理，模型来自 agent 支持模型列表。
- 首次进入会话主页时，如果运行模式配置不完整，优先引导用户配置运行模式。

创建规则：

- 标题 = 输入内容首行前 12 个 Unicode 字符；无文字时使用 i18n `New Task / 新任务`。
- 自动标题只在第一次创建时生成；生成完成后用户才能手动改名，手动标题不再被自动覆盖。
- description 为空，requirement/content 是完整输入。
- 提交前先校验 workspace、运行模式、agent、模型、AUTO 设置、workflow 模板有效性。
- 校验通过：创建 task + run，并进入该 run 的当前 session。
- 校验失败：展示缺失项和恢复路径，例如跳转 Run Mode Management、Agent 管理、上下文管理或选择模型。
- 初始附件绑定 task，作为原始 task 输入的一部分；重跑复用这些附件。
- 继续会话时新增附件进入当前 ACP session，可先通过 ACP 传给 agent，不额外做 task 级持久化。
- task/run 创建后不可再移动 workspace。

### Runtime window

- 左侧选中最小单位是 run。
- 右侧主区域永远显示当前/选中 session 的具体对话，不做完整 run 时间线主视图。
- session switcher 放在会话窗口顶部信息展示区，显示路径如 `round-001/dev/attempt-002`。
- 点击 switcher 后展示 round → node → attempt/session 的层级树，用户可切换具体 session。

默认 session 选择：

- 如果用户已有选中 session 且仍有效，保持选中。
- 有多个 session 时默认最近 session。
- run 结束时显示到达 end 状态的 session；若无法判定，则回退到最近 terminal session。

自动切换规则：

- 上一个 session 完成，且消息窗口在底部时，才自动切换和折叠历史。
- 用户不在底部表示正在看历史，不自动切换、不折叠。
- 只有一个 session 正在运行时，自动展开该 session。
- 多个 session 正在运行时，在主对话上方显示几个折叠行，展示 session 名称和实时状态；用户点击后进入该 session。
- 多个运行 session 中完成的行消失；若只剩一个运行 session，则按自动展开规则处理。

Composer 与顶部操作：

- 输入框内状态互斥：正常输入、继续按钮、工作流无效修复按钮、运行错误提示/操作。
- `continueRun` 继续同一个 session，不再进入工作流 runtime；复用现有实现。
- 停止并重跑是顶部操作区按钮，常显。
- 当前 run 运行中点击重跑：弹窗二次确认，确认后停止当前 run 并创建新 run。
- 当前 run 已结束点击重跑：直接创建新 run。
- run 历史始终保留，重跑只新增 run 并切换过去。
- WORKFLOW 模式显示编辑工作流 icon，复用现有 workflow drawer；修改只影响未来 run，不影响当前 run snapshot。
- composer 下方显示当前 session 的产物和附件信息区，有内容才显示，点击查看详情。

### Run Mode Management

新增运行模式管理页，只管理：

- AUTO 设置
- 工作流模板管理

不合并：

- Agent 管理
- 上下文管理

AUTO 本质上生成一个只有 AI-DYNAMIC 节点的工作流，因此仍复用现有 workflow validation、snapshot、runtime 执行能力。每个 workspace 记忆上次选择的运行模式。

## Data / Storage Plan

### Config and state

在 `src/config/mod.rs` 的状态/设置结构中新增独立的新 UI 状态，避免污染旧工作台字段：

- `desktop_ui_mode`: `conversation | workbench`
- `conversation_workspaces`: 多 workspace 列表
- `last_conversation_workspace`: 新会话默认 workspace
- `conversation_pins`: 有序置顶 task refs
- `conversation_run_modes`: workspace 级运行模式记忆与 AUTO 设置

推荐使用 `projectId` 作为前端 URL 与会话引用标识，避免 workspace path 进入 URL，也避免跨 workspace task id 冲突。后端提供 `projectId -> workspacePath` resolver。

### Conversation metadata

为会话 UI 创建的 task 增加 task 级 metadata，例如：

`~/.gold-band/projects/{project_id}/tasks/{task_id}/authoring/conversation.json`

内容：

- `version`
- `source: "conversation-ui"`
- `runMode: "auto" | "workflow"`
- `workflowTemplateId`
- `autoConfigSnapshot`
- `initialAttachmentNames`
- `titleAutoGenerated`
- `createdAt`

旧 task 没有该文件时仍可显示，但以兼容展示为主；能找到最新 run 就打开 latest run，找不到可显示最小恢复态并提供打开工作台入口。

### Task-level attachments

新增 task 初始附件目录：

`~/.gold-band/projects/{project_id}/tasks/{task_id}/authoring/attachments/`

创建时上传的附件写入此目录，run 启动时复制或引用到首个 attempt 的附件上下文。重跑复用 task-level attachments。

### SQLite search

`src/storage/sqlite.rs` 已有 `tasks` / `sessions` / `session_prompts` 与 FTS。新 UI 搜索只搜 task/conversation 记录，并跨 workspace。

实施前需要修正多 workspace 索引键：

- 现有 tasks 表以 `task_id` 为主键，跨 workspace 可能冲突。
- 调整为 `task_path` 主键，或增加 `(project_id, task_id)` 唯一键。
- 给搜索结果增加 `projectId` / workspace 展示信息。
- SQLite 是辅助索引，可通过 schema version bump + full backfill 处理。

## Backend / Tauri API Plan

在 `src-tauri/src/commands.rs` 增加命令，`src-tauri/src/main.rs` 注册，并在 `web/src/api/client.ts`、`web/src/api/desktop.ts`、`web/src/api.ts`、`web/src/api/browser.ts` 暴露。

### New commands

- `save_desktop_ui_mode(mode)`：持久化 `conversation | workbench`。
- `get_conversation_sidebar()`：返回 workspace 列表、置顶列表、按 workspace 分组的 task rows、最新 run summary。
- `choose_conversation_workspace()`：选择并加入会话 UI workspace 列表，更新默认 workspace。
- `get_conversation_task_runs(projectId, taskId)`：展开 task 行时获取 run 列表。
- `search_conversation_tasks(query, limit)`：包装现有 `search_tasks`，返回 projectId、workspace、task、latestRun 等富结果。
- `get_conversation_run(projectId, taskId, runId, selectedSessionKey?)`：返回 run 级 session tree、选中 session、active sessions、task/run 摘要、artifacts/attachments、workflow 状态。
- `validate_conversation_create(input)`：创建前校验并返回缺失项与恢复路径。
- `create_conversation_run(input)`：创建 task、写 conversation metadata、保存初始附件、启动 run、返回 conversation run VM。
- `rerun_conversation_task(projectId, taskId, currentRunId?, stopActive)`：必要时停止当前 active run，再创建新 run。
- `update_task_metadata(projectId, taskId, title, description?)`：保存标题修改并重建/更新 SQLite 索引。
- `pin_conversation` / `unpin_conversation` / `reorder_pinned_conversations`。
- `get_conversation_run_mode(projectId)` / `save_conversation_run_mode(projectId, settings)`。

### Reuse existing backend code

- `create_task` / task authoring persistence from `src/app/mod.rs` and `src-tauri/src/commands.rs`。
- `start_run` / `continue_run` / `kill_run` from existing runtime commands。
- `WorkflowTemplateStore` CRUD from `src/app/mod.rs` and `commands.rs`。
- workflow validation and agent doctor checks, especially `ensure_workflow_agents_doctor_ready`。
- `run_summary_vm`、`round_summary_vm`、`acp_session_vm`、`dynamic_acp_session_vm` patterns in `src-tauri/src/view_models.rs`。
- SQLite `search_tasks` / index retry helpers from `src/storage/sqlite.rs`。

### Run-level session aggregation

在 `src-tauri/src/view_models.rs` 新增 run 级会话 VM 构造逻辑：

- 遍历 run 下所有 round / node / attempt。
- 对 AI-DYNAMIC 动态节点遍历 dynamic attempt 目录，不只看 latest。
- 每个具体 ACP session 作为 leaf：`roundId`、`nodeId`、`attemptId`、`outerNodeId`、`outerAttemptId`、`pathLabel`、`status`、`outcome`、`current`、`startedAt`、`finishedAt`、`sessionId`、`artifactCount`、`attachmentCount`。
- `get_conversation_run` 只加载选中 session 的完整 `AcpSessionVm`，不要一次加载全 run 所有 events。

### AUTO workflow generation

AUTO 模式在创建 task 前生成标准 `WorkflowDsl`：

- 一个 `ai-dynamic` 节点作为 entry。
- 成功 edge 指向 end。
- agent、model、permission、allowed profiles、global goal、control limits、allowed workflows 来自 workspace 级 AUTO 设置。
- 生成后走现有 workflow validation、snapshot、runtime。

### Agent model list

在 `ManagedAgentVm` 中增加 `supportedModels?: AcpModelVm[]`。

- 优先从 provider doctor capabilities 读取。
- 不支持模型列表的 agent 返回空数组；是否必选模型由 agent capability/配置决定。
- 如果模型会影响运行，需要在 node resolved config / ACP invocation 中传入 selected model。

### Runtime actions

`ConversationRunPage` 顶部操作：

- title inline edit → `update_task_metadata`。
- rerun → `rerun_conversation_task`；active run 先 `AlertDialog` 确认。
- workflow edit icon → 复用 `WorkflowPage.tsx` 的 workflow drawer + `WorkflowEditor`。
- graph/workflow icon 可打开现有 `GraphView` 视图用于查看 runtime graph。

Composer 状态：

- invalid workflow → 修复按钮打开 workflow drawer 或 Run Mode Management。
