## 任务 
我需要实现以下需求：
1、当客户端启动时，向服务器发送心跳，心跳接口为：http://maling.weoa.com/api/client-report/heartbeat
2、当用户使用客户端发起任务时，每完成一个节点，向服务器发送该节点的详细信息，节点详情接口为：http://maling.weoa.com/api/client-report/metrics/batch

## 具体流程

### 渠道配置（configs/channels）流程
1、设置页面做一个开关，开关打开，需要填写指标上报地址；
2、configs/channels下新增四个配置：开关默认状态、开关是否允许切换、指标上报服务根地址、API Key
3、default版本：开关默认关闭，允许切换，指标上报服务地址为空，打开的话需要自行填写服务根地址。
4、wb版本：开关默认打开，开关是不允许切换，指标上报服务地址默认：http://maling.weoa.com，禁止修改；心跳与节点详情两个接口后缀由客户端内置协议常量拼接，不在设置页展示。

### 发送心跳功能流程
1、客户端启动以后，设置定时任务，每五分钟调用一次，调用心跳接口
2、心跳接口参数为json格式 ，参数如下：
public class ClientHeartbeatReportForm {

    @NotBlank(message = "userId is required")
    @Size(max = 128, message = "userId length must be <= 128")
    private String userId;

    @Size(max = 255, message = "workspace length must be <= 255")
    private String workspace;

    @Size(max = 64, message = "clientVersion length must be <= 64")
    private String clientVersion;

    /** Optional client-side report time; server fills with now() if null */
    private LocalDateTime reportedAt;

}
3、userId字段取值：windows系统取当前用户名（powershell可获取：$env:USERNAME；CMD:echo %username%），IOS系统使用bash：echo $USER
   workspace字段取值：取当前工作空间
   clientVersion：当前客户端版本号
   reportedAt：取当前时间格式为：YYYY-MM-DD HH:MM:SS
   
### 发送节点详情接口流程
1、客户端每一个节点开始执行的时候，需要同时发送前序节点和当前节点详细信息给节点详情接口
2、节点详情接口参数为json格式，参数如下：
@PostMapping("/api/client-report/metrics/batch")
@Data
public class ClientNodeMetricBatchReportForm {

    @NotEmpty(message = "metrics must not be empty")
    @Size(max = 1000, message = "metrics size must be <= 1000 per request")
    @Valid
    private List<ClientNodeMetricReportItem> metrics;
}
public class ClientNodeMetricReportItem {

    @NotBlank(message = "workspace is required")
    @Size(max = 255)
    private String workspace; //当前工作空间

    @NotBlank(message = "userId is required")
    @Size(max = 128)
    private String userId; //取值同心跳

    @NotBlank(message = "taskId is required")
    @Size(max = 128)
    private String taskId; //任务ID

    @NotBlank(message = "runId is required")
    @Size(max = 128)
    private String runId; //任务runId

    @NotBlank(message = "roundId is required")
    @Size(max = 128)
    private String roundId; //任务的roundId

    @NotBlank(message = "nodeId is required")
    @Size(max = 128)
    private String nodeId; //任务的nodeId

    /** Sequence number of the node within the round */
    private Integer seq; //任务当前RUN是第几次执行
	
	private String nodeName; //节点的角色名称，取DSL中节点的profile name

    @Size(max = 64)
    private String agentType; //节点使用agengt(如claude-acp)

    @Min(value = 0, message = "attemptCount must be >= 0")
    private Integer attemptCount; //当前节点的重试次数

    private LocalDateTime startedAt; //节点详情页面的开始时间
    private LocalDateTime endedAt; //节点详情页面的结束时间
	
	@Min(value = 0, message = "inputTokens must be >= 0")
    private Long inputTokens; //节点详情页面输入Token

    @Min(value = 0, message = "outputTokens must be >= 0")
    private Long outputTokens; //节点详情页面输出Token

    @Min(value = 0, message = "cacheReadTokens must be >= 0")
    private Long cacheReadTokens; //节点详情页面缓存读取Token

    @Min(value = 0, message = "totalTokens must be >= 0")
    private Long totalTokens;  //节点详情页面总计Token

    /** Node status — must be one of RUNNING / SUCCESS / FAILED / Reentrancy */
    @NotBlank(message = "status is required")
    @Size(max = 32)
    private String status; //节点状态，开始执行时为RUNNING，执行失败为FAILED，节点执行成功为SUCCESS，同一nodeId第二次进入执行时为Reentrancy（通过attempt递增判断：attempt-001为RUNNING，attempt-002起为Reentrancy）

    /** Optional client-side report time; server fills with now() if null */
    private LocalDateTime reportedAt; //上报时间
}
3、字段取值说明在注释里
4、节点进入执行前需要判断是否有前序节点，如果没有前序节点，默认前序节点为开始节点（nodeName为开始），其他详情字段都可为空，metrics始终两条数据，前序节点数据和当前节点数据
5、taskId，runId，roundId，nodeId这些字段取值的时候不应该去页面中展示的值，页面在创建任务，新建RUN，新建round，节点编排的时候，除了页面展示的ID字段，还要生成一个隐藏id字段，使用UUID，给当前任务使用


### 渠道配置新增字段汇总

configs/channels 下每个渠道 JSON 新增以下 4 个字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `metricsEnabled` | boolean | 指标上报开关默认状态 |
| `metricsToggleLocked` | boolean | 开关是否允许用户切换（true时UI中disabled） |
| `metricsBaseUrl` | string | 指标上报服务根地址，客户端按内置后缀拼接实际接口 |
| `metricsApiKey` | string | API Key，通过请求头 `X-Maling-Report-Key` 传递 |

**default 渠道**：`metricsEnabled: false`，`metricsToggleLocked: false`，服务根地址和 Key 为空，用户打开开关时需自行填写。

**wb 渠道**：`metricsEnabled: true`，`metricsToggleLocked: true`，服务根地址预填为 `http://maling.weoa.com` 且禁止修改。

实际上报接口由客户端统一拼接，设置页隐藏两个后缀：
- 心跳：`/api/client-report/heartbeat`
- 节点详情：`/api/client-report/metrics/batch`

### API Key 管理策略

- API Key 内置在渠道配置文件中，不硬编码到源码
- wb 渠道在 `wb.json` 中直接内置 Key
- default 渠道用户需在设置页手动填写 Key
- 客户端从 channel config 读取 Key，通过 `X-Maling-Report-Key` 请求头发送
- 禁止将 Key 写入任何日志

### 补充说明

1. **status 枚举值**：`RUNNING / SUCCESS / FAILED / Reentrancy`（无 PAUSED）。Reentrancy 表示同一 nodeId 第二次进入执行，通过 attempt 递增判断。
2. **nodeName 取值**：取 DSL 中节点的 profile name。
3. **工作空间切换**：切换工作空间不影响心跳，心跳保持原有定时逻辑，不重发。
4. **前置依赖**：客户端需引入 HTTP 客户端库（如 reqwest），当前项目无 HTTP 请求能力。
5. **ID 改造**：taskId / runId / roundId / nodeId 各增加一个隐藏的 UUID 字段供上报使用，页面仍展示原有编号格式。

### 关键架构决策
- **配置层**：参考 updaterEndpoint 模式，channel config 存服务根地址默认值，SettingsConfig 可运行时覆盖；两个接口后缀由 `src-tauri/src/metrics.rs` 常量统一管理
- **跨 crate 通信**：通过 `App::metrics_callback`（Arc 回调）实现 library crate → src-tauri 的桥接
- **节点指标**：orchestrator 在 node_started 时发送（前序+当前），在 node_completed 时更新 RunState.last_executed_node
- **前序节点追踪**：通过 RunState.last_executed_node 跨 Round 追踪
- **UUID**：Option<String> + #[serde(default)]，旧数据不迁移
