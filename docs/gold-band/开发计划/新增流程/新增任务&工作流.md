1.首页任务列表，去掉导入requirements按钮，只保留创建任务
2.点击创建任务按钮，弹出创建任务弹窗
3.弹窗中支持手动填写任务标题、描述和详细需求内容，也支持导入现有需求（支持txt\md格式）覆盖当前编辑内容
4.填写或导入需求后，需要创建工作流
5.工作流可以新增节点和边，为每个节点配置不同的agent，agent来源为agent管理页面的agent卡片
6.工作流保存后持久化为json，在任务详情页的工作流中可以修改工作流，round详情中的工作流则是每一次工作流的运行时快照
7.节点前后需要有环境的生成和销毁(比如释放agent等)

关于工作流的修改与新增：最好是能做到画布式拖拽生成工作流，是否有现成组件可以使用？
关于工作流中的节点与边：
现有的节点类型现在可能要去掉execute节点，和execute-plan产出节点

节点的本质类型只有三种。1.普通节点 2.是否需要人工check？（人工check逻辑你先留白，本期不实现） 2.是否需要验证AI输出结果？ 2.1 输出的DSL配置（也就是AI要输出的指定类型，目前支持json） 2.2 DSL成功条件（json中指定字段的值） 2.3 成功流向（去哪个节点） 2.4 失败流向（去哪个节点），边可以设置一些属性，比如成功边，失败边，边可以指向其他节点（则节点生成新的attempt），或者new round（则生成新的round），边指向节点时，如果是回路（即A-C--》D--》E-->...-->->B-->A）,在B节点指向A的边中可以提供一个选择，是开启新的session，还是继续A刚刚的session？（这里涉及到环判断，可能比较复杂，如果实现困难，可以先用简单方式，即验证输出结果类型的节点，失败边提供这两个选项即可）
工作流这里可以提供一套默认工作流给用户选择：
plan（普通节点，方案角色） --> 开发（普通节点，开发角色） --> review（普通+验证输出结果，DSL中输出review是否通过，成功指向测试，失败指向开发，默认是继续开发原session） --> 测试（普通+验证输出结果，DSL中输出test是否通过，默认是继续开发原session） --> 验收（普通+验证输出结果，DSL中输出verify是否通过） --> cleanup（普通节点，清理角色，无 AI 输出验证） --> 结束

## MVP 实现状态

已实现：
- 任务列表移除独立“导入 requirements”入口，只保留“创建任务”。
- 创建任务 Sheet 采用编辑态，支持填写 title / description / requirement 正文，并可选导入 `.txt` / `.md` requirement 覆盖当前编辑内容；保存任务入口固定在 Sheet 标题栏右侧。
- 工作流编辑器基于 `@xyflow/react`，支持通过画布左上角浮动工具条新增/删除节点、连接边、配置边类型、目标与 session 策略；画布右键菜单可按需添加“结束”和“新 Round”两个虚拟目标节点，普通节点可直接拖拽连线到它们；节点位置由系统自动布局，新增节点后自动聚焦，右侧 Inspector 只承载属性配置。
- 节点 id 输入采用本地草稿，中文输入法组合输入期间不实时改写 workflow DSL，失焦、Enter 或组合结束后再提交；作者态画布普通节点直接展示原始节点 id，不对 `test` 等名称做本地化替换。
- 作者态画布允许编辑过程中临时存在多条同类型出边，便于先拖拽连线再修改边类型；创建任务 Sheet 标题栏的保存任务按钮与模板保存/另存按钮都会校验同一来源节点的同一结果类型只能有一条出边，任务详情修改 workflow 时由保存工作流按钮触发同一校验。所有回退边都会做 lane 路由，review/test 回 dev 时不与主成功路径或其他回退边重合。
- 节点 Agent 来源为 Agent 管理页已配置且支持的 agent card；前端不提供默认 provider，新增节点必须显式选择可用 Agent；节点 profile 字段在中文界面显示为“角色”，英文界面显示为“Profile”。
- 默认工作流为 `plan -> dev -> review -> test -> accept -> cleanup -> $end`，默认不再生成 `worker` 节点和 `节点输出产物` 产物；默认模板来自后端持久化内置 workflow JSON，前端不再维护本地默认 workflow 生成器。默认模板的 `control.max_attempts` 与 `control.max_rounds` 留空，表示默认不限制。
- 默认角色由客户端内建资源直接提供方案、开发、审查、测试、验收、清理六个角色，再把对应 profile `id` 绑定到默认 workflow 节点；cleanup 是普通 worker 节点，不启用 AI 输出验证。
- 内置角色与默认工作流模板保持一致：可在 UI 中打开并编辑草稿，但不能直接保存覆盖，只能另存为新的普通角色；删除角色时若仍被 workflow 模板、任务 authoring workflow 或可继续/可重试/人工 check 的运行时快照引用，首次点击删除会在弹窗内提示引用范围以及删除后需要修复关联 workflow / snapshot，用户二次确认后仍可删除。
- 默认 review/test/accept 使用简化 AI 输出结构 `{"reason":"String","result":"boolean"}` 与 `$.result == true` 成功表达式；前端默认模板按钮应用后端模板，不维护独立默认 schema/expression。
- worker 节点支持 JSON 输出验证：`output.kind=json` + 简化 `output.schema` + `success_condition.expression`，并兼容旧 `success_condition.path/equals` 条件形式；旧完整 JSON Schema 不再兼容。
- JSON 输出约束输入采用本地草稿：停止输入约 2 秒或失焦后再校验并写入 workflow DSL；格式化改为输入框右上角悬浮按钮手动触发，避免输入中途被自动 beautify 打断。字段问号说明以及 profile 标签旁帮助统一使用圆形问号 icon + 随主题变化的浅色 shadcn/ui `Tooltip`，悬浮或聚焦即可出现；原本带明确语义的其他说明 icon（如 profile summary）保持原语义，不强行替换成问号。
- worker 节点的 AI 输出验证与人工 check 是二选一的结果判定方式，开启其中一种会自动关闭另一种。
- 保存 workflow 时前端会先聚合校验必填字段、角色绑定、角色可见性、成功表达式路径，以及当前 `workflow.id` 是否与模板库内其他模板重复；错误以一次弹窗集中展示，关闭后错误字段红色高亮并显示原因；后端仍会在保存时做同样业务约束的最终裁决，并通过 `code + params` 返回结构化错误，由前端 i18n 负责可见文案。
- 边可指向普通节点、`$end` 或 `$new-round`；工作流至少需要一条边最终指向 `$end`，且所有节点都必须从入口节点可达，否则不能保存。指向普通节点时会生成新 attempt，指向 `$new-round` 时打开新 round；`success` 边不能指向 `$new-round`，作者态目标下拉会隐藏该选项，保存与 DSL 校验都会拦截。
- 节点会话按边上的 session 策略展示：`session=new` 的 attempt 始终是独立可切换 conversation；`session=continue` 挂回被继续的 provider session，并在同一 conversation 内用 attempt 分隔行展示。
- 工作流控制项支持可选的 `max_attempts` 与 `max_rounds`：前者在单个 round 内限制 `failure` 触发的修复/重试跳转次数，正常 `success` 前进和 schema 隐藏修复不消耗次数；后者限制 `$new-round` 打开的新 round 次数；留空表示不限制，默认模板也采用留空不限，超限时 workflow 失败并展示结构化停止原因。
- 任务级 workflow 保存到 `authoring/workflow.json`，run 启动时冻结为 `workflow.snapshot.json`；Round 详情展示运行态快照，不回写后续作者态修改。
- 用户级 context 存储维护跨 workspace 工作流模板列表 `~/.gold-band/context/workflows.json`，通过 `templates` 数组支持多个命名 workflow，并记录上一次创建任务使用的 workflow；创建任务 Sheet 中上次使用的命名模板以“上次创建使用的是：xxx”提示展示在下拉框旁，点击模板名后选中该模板并隐藏提示，不作为下拉选项；模板下拉顶部支持新增模板并进入空白画布，行内支持删除非默认模板，修改非默认模板后支持直接保存修改到当前模板，默认模板和空白画布只允许另存为新模板；模板保存/删除成功提示短暂展示后自动消失，错误提示展示在工作流模板操作区下方并持续保留，不占用创建任务顶部全局位置。
- “另存为新的工作流模板”时后端会自动重置 `workflow.id` 为新的分布式唯一值，不向用户暴露内部碰撞；如果用户手动编辑 JSON 导致模板库内 `workflow.id` 与其他模板重复，则保存模板会失败并返回 `workflow.duplicate-id`。
- 工作流编辑器 JSON 视图支持直接编辑完整 workflow DSL，便于用户手动修复 `workflow.id` 冲突等模板级问题。
- AI-DYNAMIC 保存时会额外校验其引用的所有模板；只要被引用模板存在 `workflow.id` 与其他模板重复的脏数据，当前工作流就不能保存，直到用户修复对应模板 JSON。
