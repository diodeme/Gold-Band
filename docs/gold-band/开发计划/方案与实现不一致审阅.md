# Gold Band 方案与实现不一致审阅

## 结论

当前产品设计文档、现有代码实现、以及 `功能点todo列表.md` 三者之间，主链路方向基本一致，但仍存在一批需要单独收敛的不一致点。这些不一致主要集中在 CLI 契约、状态文件 schema、provider/session 边界、verify 证据选择、以及部分 DSL / control 默认行为上。当前 console 交互层已完成一次 workflow-first 重构，旧的 explorer-first 设计不再作为当前实现基线。

---

## 1. CLI 参数契约不一致

### 1.1 `run` 命令参数模型
- 方案文档示例更偏向：
  - `run status <run-id>`
  - `run continue <run-id>`
  - `run retry <run-id>`
  - `run kill <run-id>`
- 当前代码实际要求：
  - `task_id + run_id`

### 方案链接
- `docs/gold-band/产品设计文档/interaction/cli.md:61-72`

### 影响
- 文档使用者会误判 CLI 入参
- 后续插件或脚本层若按文档接 CLI，会直接不兼容

---

## 2. `open-session` 行为语义不一致

### 2.1 文档语义
方案文档更偏向“由 CLI 直接打开 provider 原始会话 / 新终端 handoff”。

### 2.2 当前实现
当前实现更接近“输出 provider-native command”，而不是实际打开会话。

### 方案链接
- `docs/gold-band/产品设计文档/interaction/cli.md:111-121`
- `docs/gold-band/产品设计文档/provider/worker-ref.md:76-93`

### 影响
- `run open-session` 是否算“已完成”在产品语义上有歧义
- CLI、插件、用户心智模型不统一

---

## 3. 状态文件 schema 命名风格不一致

### 3.1 文档侧
文档中的 `task.json`、`run.json`、`node.json`、`worker-ref.json` 示例与字段说明使用 camelCase。

### 3.2 当前实现侧
当前代码落盘更偏向 Rust 默认序列化风格，实际字段命名与文档 schema 不完全一致。

### 方案链接
- `docs/gold-band/产品设计文档/runtime/state/task.json.md:15-43`
- `docs/gold-band/产品设计文档/runtime/state/run.json.md:16-50`
- `docs/gold-band/产品设计文档/runtime/state/node.json.md:16-53`
- `docs/gold-band/产品设计文档/provider/worker-ref.md:30-45`

### 影响
- 文档无法直接作为落盘 contract 使用
- 插件/外部工具若按文档读取状态文件，会有字段不兼容风险

---

## 4. `worker-ref.json` 结构不一致

### 4.1 文档侧
文档中 `worker-ref.json` 倾向于：
- camelCase 字段
- `openCommand` 为结构化对象
- 更完整的字段合法性约束

### 4.2 当前实现侧
当前实现是最小结构，且校验强度更弱。

### 方案链接
- `docs/gold-band/产品设计文档/provider/worker-ref.md:30-69`

### 影响
- 当前只能说“最小可用”，还不能说已完全符合设计 contract
- 后续 provider 扩展时容易把 runtime 边界做散

---

## 5. prompt bundle 设计强于现有实现

### 5.1 文档侧
设计文档中 prompt bundle 期望包含更完整的：
- role contract
- artifact schema prompt
- cold attachment index
- 更清晰的 runtime context

### 5.2 当前实现侧
当前实现已经有基础 system/user prompt 与冷数据索引，但还未完全达到文档中的完整契约。

### 方案链接
- `docs/gold-band/产品设计文档/provider/prompt-bundle.md:97-243`
- `docs/gold-band/产品设计文档/provider/implementations/claude-code.md:71-145`

### 影响
- todo 列表里“prompt bundle 渲染”如果标为完成，会显得偏乐观
- 后续 artifact contract 稳定性可能不够

---

## 6. DSL 校验设计强于现有实现

### 6.1 文档侧
DSL 文档定义了比当前代码更完整的校验集，包括：
- loop 参数正整数校验
- 无 verify 时 acceptance 配置非法
- 保留字冲突
- `$end` 合法性
- `session=continue` 的目标能力校验

### 6.2 当前实现侧
当前只实现了基础结构校验与部分 continue / planFrom 约束。

### 方案链接
- `docs/gold-band/产品设计文档/dsl/control.md:316-350`

### 影响
- 现在的 DSL “基本可用”，但还不是“设计完整收敛版”
- todo 列表里“Workflow DSL 基本校验完成”需要配合新增细项一起理解

---

## 7. `exec.invalid` 默认 repair 行为不一致

### 7.1 文档侧
文档定义：
- `exec.invalid` 若无显式 edge，应默认回到 `planFrom` 对应 worker
- 优先尝试 `continue`
- provider 不支持再降级 `new`

### 7.2 当前实现侧
当前实现更偏向直接进入阻塞暂停。

### 方案链接
- `docs/gold-band/产品设计文档/runtime/control.md:215-220, 412-413`
- `docs/gold-band/开发计划/gold-band-mvp-plan.md:316-320`

### 影响
- 这会直接影响 repair loop 的产品行为
- 是当前最需要明确收敛的控制流差异之一

---

## 8. verify 证据选择与 attachment 暴露不一致

### 8.1 文档侧
verify 证据包设计更完整，要求关注：
- 当前 round 最新 `exec-result`
- 当前 round 最新上游 worker primary artifact
- 显式暴露的 attachments

### 8.2 当前实现侧
当前实现已具备基础证据选择，但 attachments 与更严格的“最新上游 worker artifact”规则尚未完全落地。

### 方案链接
- `docs/gold-band/产品设计文档/provider/invocation.md:173-188, 206-213`
- `docs/gold-band/产品设计文档/runtime/layout.md:408-425`

### 影响
- verify 的输入边界还不够稳定
- acceptance loop 的反馈质量会受影响

---

## 9. `exec` attempt layout 不完全一致

### 9.1 文档侧
`exec` attempt layout 中明确包含：
- `exec-plan.source.json`

### 9.2 当前实现侧
当前主要落 canonical `exec-result.json` 与命令执行 sidecar，`exec-plan.source.json` 仍缺。

### 方案链接
- `docs/gold-band/产品设计文档/runtime/layout.md:339-355`

### 影响
- attempt 目录的可追溯性低于设计目标
- exec 输入来源的审计信息不完整

---

## 10. `RuntimeConfig.default_provider` 与用户目录解析未完全收敛

### 10.1 文档侧
文档明确把 `default_provider` 视为 runtime 配置核心项，并把用户目录定义为 `~/.gold-band/`。

### 10.2 当前实现侧
当前默认 provider 的配置生效链路还不完整；用户目录解析也更接近固定路径，而不是 home 语义。

### 方案链接
- `docs/gold-band/产品设计文档/runtime/overview.md:81-92`
- `docs/gold-band/产品设计文档/runtime/layout.md:48-68`

### 影响
- 配置能力和跨机器可移植性都还不完整

---

文档中的 stage 枚举采用下划线风格示例。

### 11.2 当前实现侧
当前实现的序列化命名风格与文档未完全一致。

## 12. `inspect` / `provider` / `index.json` 仍属于设计弱定义区

### 12.1 当前情况
这些能力在文档中已有方向，但仍缺：
- 输出 schema
- 错误模型
- 稳定读取契约

### 方案链接
- `docs/gold-band/产品设计文档/interaction/cli.md:130-145`
- `docs/gold-band/产品设计文档/runtime/layout.md:145-150`

### 影响
- 这些模块还不适合直接判断为“方案已覆盖完整”

---

## 收敛建议

建议按以下顺序收敛：

1. 先统一 **状态文件 / worker-ref / progress** 的 schema contract
2. 再统一 **CLI 参数与 open-session handoff** 的产品语义
3. 再补齐 **DSL 校验、exec.invalid 默认 repair、verify 证据边界**
4. 最后补 **inspect/provider/index.json/progress.events** 这些扩展模块文档
