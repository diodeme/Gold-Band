# Gold-Band 前后端性能分析优化报告

本报告基于当前代码静态扫描、已安装的性能相关 skills 方法论整理，包括 `performance`、`memory-leak-debugging`、`memory-leak-audit`、`m10-performance`、`rust-async-patterns`、`rust-profiling`、`tauri`。本报告不是实测 profiling 报告；结论优先级来自代码路径、数据规模放大风险、Tauri IPC/文件 I/O 特征和前端渲染成本判断。

## 1. 总体结论

当前项目的主要性能瓶颈不在单个 React 组件或某个 Rust 函数，而在以下系统性路径：

1. **ACP 会话链路的 JSONL 读写与轮询**
   - 前端 1.5s 轮询 ACP session。
   - 后端每次构建 ACP session VM 时扫描事件、诊断、raw frame 文件。
   - 随着会话时长增长，I/O 和 CPU 成本线性上升。

2. **raw frame 面板前后端双重重活**
   - 后端查询 raw frame 时会全量扫描甚至双遍扫描。
   - 前端每行 raw frame render 时立即 `JSON.parse + JSON.stringify + pretty print + wrap`，即使用户没有展开。

3. **任务/Workflow 列表的文件系统 N+1 遍历**
   - task list、workflow、round detail 会反复读取状态文件、遍历 run/round/node/attempt/artifact。
   - 全局 10s background refresh 会放大这个成本。

4. **前端 ACP 聊天从虚拟列表改为手写滚动后，风险可控但上限变低**
   - 当前通过 `EVENT_WINDOW_LIMIT = 360` 限制 DOM 数量。
   - 没有虚拟化后，360 条复杂消息一次性渲染仍可能卡顿，尤其 tool/thought/raw-heavy 内容。

5. **worker 执行链路存在内存与稳定性风险**
   - `Command::output()` 会把 stdout/stderr 全量读入内存。
   - `timeout_sec` 当前未实际生效。
   - 长输出或挂起命令会影响整体稳定性。

## 2. 优先级矩阵

| 优先级 | 问题 | 影响维度 | 建议处理阶段 |
| --- | --- | --- | --- |
| P0 | ACP session 轮询触发后端 JSONL 全量扫描 | 响应、CPU、I/O、内存 | 立即优化 |
| P0 | raw frame 后端双遍扫描 + 前端即时 pretty print | 响应、CPU、内存、I/O | 立即优化 |
| P0 | worker stdout/stderr 全量进内存，timeout 未生效 | 内存、稳定性、响应 | 立即优化 |
| P1 | task/workflow summary N+1 文件遍历 | 响应、I/O、CPU | 近期优化 |
| P1 | ACP 手写滚动无 rAF 节流，360 条复杂 DOM | CPU、GPU、交互响应 | 近期优化 |
| P1 | GraphView dagre layout 频繁重算 | CPU、交互响应 | 近期优化 |
| P2 | 未使用依赖 `react-virtuoso` 仍保留 | 依赖体积、维护成本 | 顺手处理 |
| P2 | 系统字体每次全量扫描 | 响应、CPU | 设置页优化 |
| P2 | 小型 timeout cleanup 风险 | 内存泄漏边界 | 顺手处理 |

## 3. 前端性能分析

### 3.1 ACP 聊天主列表：去掉 `react-virtuoso` 后的影响

#### 现状

新版 `ACPChatDialog` 已不再使用 `Virtuoso`，改为手写滚动容器。

相关位置：

- `web/src/components/acp/ACPChatDialog.tsx:36-39`
- `web/src/components/acp/ACPChatDialog.tsx:453-467`

关键限制：

```ts
const EVENT_WINDOW_LIMIT = 360;
const EVENT_PAGE_SIZE = 60;
```

渲染位置：

- `web/src/components/acp/ACPChatDialog.tsx:461`

```tsx
{timeline.map((event) => (
  <div className="px-5 py-2" key={timelineEventKey(event)}>
    <ACPEventRenderer event={event} onLayoutChange={preserveScrollPosition} />
  </div>
))}
```

#### 性能判断

优点：

- 事件窗口有限，不是无界 DOM。
- 360 条以内普通文本消息通常可接受。
- 去掉虚拟化后，滚动锚点逻辑更可控，不依赖第三方列表行为。

风险：

- 360 是事件数，不是 DOM 节点数。一个 tool card / thought block 可能展开成很多 DOM。
- 每次 timeline 更新会让整个 map 参与 React reconcile。
- 如果 tool output 很大，展开后仍可能卡。
- 长会话下靠窗口裁剪隐藏问题，但切历史时仍会频繁 merge/sort/build timeline。

#### 建议

1. **把聊天窗口默认从 360 降到 180-240**
   - 对桌面聊天体验通常足够。
   - 老历史通过 `loadOlderEvents` 分页加载即可。

2. **对复杂消息做 collapsed-first 渲染**
   - `ToolCallCard` 默认只展示 summary。
   - thought 默认折叠。
   - 大 output 不直接渲染完整内容，提供“查看完整 raw/log”入口。

3. **如果未来 ACP 事件量明显增长，再恢复虚拟化**
   - 可以不一定用 `react-virtuoso`，也可用 `@tanstack/react-virtual`。
   - 但当前更优先的是 raw frame 和后端 JSONL 优化。

### 3.2 ACP 手写滚动：scroll handler 未节流

位置：

- `web/src/components/acp/ACPChatDialog.tsx:280-289`

当前每次 scroll 都执行：

- 计算 `distanceFromBottom`
- 设置 `pinToBottomRef`
- `setIsAtBottom(atBottom)`
- 接近顶部时 `loadOlderEvents`

#### 风险

scroll 是高频事件，虽然 React 会合并部分更新，但 `setIsAtBottom` 在连续滚动中仍可能触发多次 render 尝试。

#### 建议

用 `requestAnimationFrame` 合并 scroll：

```ts
const scrollFrameRef = useRef<number | null>(null);

const handleChatScroll = () => {
  if (scrollFrameRef.current != null) return;
  scrollFrameRef.current = window.requestAnimationFrame(() => {
    scrollFrameRef.current = null;
    // existing scroll logic
  });
};
```

并且只有状态变化时才 `setIsAtBottom`：

```ts
setIsAtBottom((current) => current === atBottom ? current : atBottom);
```

#### 验证

- Chrome Performance 录制快速滚动 ACP 聊天。
- 观察 scroll 期间 scripting time 和 React commit 次数。

### 3.3 ACP timeline 构建重复计算

相关位置：

- `web/src/components/acp/ACPChatDialog.tsx:46-59`
- `web/src/components/acp/ACPChatDialog.tsx:199`
- `web/src/components/acp/ACPChatDialog.tsx:1029-1082`
- `web/src/components/acp/ACPChatDialog.tsx:1109-1114`

当前存在这些函数：

```ts
function timelineItemCount(events) {
  return buildAcpTimeline(events).length;
}

function prependedTimelineItemCount(previous, next) {
  const previousFirst = buildAcpTimeline(previous)[0];
  // ...
  const nextIndex = buildAcpTimeline(next).findIndex(...);
}

function removedTimelineItemCount(before, after) {
  return Math.max(0, timelineItemCount(before) - timelineItemCount(after));
}
```

#### 问题

在加载 older、新消息合并、裁剪窗口时，会重复构建 timeline。窗口为 360 时暂时可接受，但如果 raw/tool event 内容复杂，CPU 成本会上升。

#### 建议

1. 让 merge/limit 返回元信息，而不是通过重新 build timeline 推断：

```ts
type LimitedEventsResult = {
  events: AcpUiEventVm[];
  removedCount: number;
};
```

2. 或者在 `buildAcpTimeline` 后缓存 key list：

```ts
const timeline = useMemo(() => buildAcpTimeline(effectiveEvents), [effectiveEvents]);
const timelineKeys = useMemo(() => timeline.map(timelineEventKey), [timeline]);
```

3. `calculateTurnTotalSeconds` 不要每秒 sort 全量 events。

位置：

- `web/src/components/acp/ACPChatDialog.tsx:941-955`

当前每秒：

```ts
const chronological = [...events].sort((left, right) => left.seq - right.seq);
```

建议在事件变化时预计算 timing metadata，计时器只更新 now。

### 3.4 raw frame 前端渲染是当前前端最大热点

位置：

- `web/src/components/acp/ACPChatDialog.tsx:899-920`
- `web/src/components/acp/ACPChatDialog.tsx:1287-1319`

当前逻辑：

```ts
function RawFrameRow({ frame }) {
  const display = rawFrameDisplay(frame.content);
  const scrollable = isLongRawFrame(display.expanded);
  // ...
}
```

`rawFrameDisplay`：

```ts
const value = JSON.parse(content);
return {
  compact: JSON.stringify(value),
  expanded: wrapLongSegments(JSON.stringify(value, null, 2)),
};
```

#### 问题

即使 `<details>` 没有展开，也会对每条 raw frame 做：

- `JSON.parse`
- compact stringify
- pretty stringify
- long segment wrapping

raw page size 最大 200：

- `web/src/components/acp/ACPChatDialog.tsx:881-887`

如果每条 content 较大，这会造成打开 raw 面板时 CPU 和内存尖峰。

#### 建议

这是前端最值得优先优化的一项。

推荐后端返回：

```ts
type AcpRawFrameVm = {
  id: string;
  lineNumber: number;
  timestamp?: string;
  direction?: string;
  kind: string;
  preview: string;
  content?: string; // 仅详情查询时返回
  contentTruncated: boolean;
};
```

前端 collapsed 状态只显示 `preview`：

```tsx
<code>{frame.preview}</code>
```

展开时再请求或计算完整 pretty content：

```tsx
{open ? <RawFrameExpanded frame={frame} /> : null}
```

如果不想新增后端接口，至少做前端 lazy：

```tsx
const [open, setOpen] = useState(false);
const display = useMemo(
  () => open ? rawFrameDisplay(frame.content) : { compact: truncateFrameLine(frame.content), expanded: '' },
  [open, frame.content],
);
```

#### 验证

场景：

- 100 条 raw frame，每条 20KB。
- 200 条 raw frame，每条 100KB。
- 测 raw 面板首次打开时间、JS heap、scripting time。

## 4. 后端 / Rust / Tauri 性能分析

### 4.1 ACP session VM 每次构建会扫描多个 JSONL 文件

前端轮询位置：

- `web/src/components/acp/ACPChatDialog.tsx:166-191`

后端命令：

- `src-tauri/src/commands.rs:202-214`

VM 构建：

- `src-tauri/src/view_models.rs:933-988`

内部扫描：

- `scan_acp_events`：`src-tauri/src/view_models.rs:1004-1126`
- `scan_acp_diagnostics`：`src-tauri/src/view_models.rs:1196-1218`
- `count_jsonl_lines`：`src-tauri/src/view_models.rs:1385-1394`

#### 问题

每次 `get_acp_session`：

1. 读 session json。
2. 扫 events jsonl。
3. 扫 raw jsonl 统计 raw frame count。
4. 扫 diagnostics jsonl 统计 error count / last error。
5. 序列化 VM 通过 Tauri IPC 返回前端。

在 active ACP 会话中，每 1.5s 执行一次。

#### 影响

- 会话越长，扫描越慢。
- 多个 ACP drawer 或长时间运行后，I/O 压力明显。
- Tauri IPC payload 也会增大。

#### 建议

最优方案：事件推送。

ACP runtime 写入 event 后，通过 Tauri event/channel 推送增量事件到前端。

前端不再 polling：

```ts
listen('acp:event', handler)
```

后端 append event 时 emit：

```rust
app_handle.emit("acp:event", payload)?;
```

次优方案：增量 cursor 读取。

保留 command，但接口强制使用：

```ts
getAcpSession({ afterSeq, eventLimit })
```

后端维护轻量 index：

- 文件 offset
- last seq
- event count
- diagnostics error count
- raw frame count

避免每次从头读。

可接受的短期方案：

- active polling 只请求 `afterSeq`。
- diagnostics/raw count 降频，例如 10s 或面板打开时才刷新。
- raw frame count 不参与常规 session VM。

#### 验证

准备数据：

- 1k / 10k / 50k 行 `acp.events.jsonl`
- 1k / 10k / 50k 行 `acp.raw.jsonl`

测：

- `get_acp_session` p50 / p95
- 单次 IPC payload size
- 前端 active ACP 页面 CPU 占用

### 4.2 raw frame 查询后端双遍扫描

位置：

- `src-tauri/src/view_models.rs:1409-1442`
- `src-tauri/src/view_models.rs:1444-1456`
- `src-tauri/src/view_models.rs:1458-1486`
- `src-tauri/src/view_models.rs:1488-1509`

当前流程：

1. `count_matching_raw_frames` 全量扫描，计算 total。
2. `collect_matching_raw_frames` 再次扫描，收集当前页。
3. 如果有 kind/direction 过滤，每行调用 `raw_frame_meta`，解析 JSON。

#### 问题

这是 O(2N) 文件扫描。搜索时还会对每行 `to_lowercase()`：

```rust
if !line.to_lowercase().contains(search)
```

#### 建议

短期：

1. 单遍扫描同时 count 和保留目标窗口。
2. 搜索 query 预先 lowercase，line lowercase 尽量避免重复分配。
3. page 只支持 latest/older cursor，避免 total 精准统计。
4. latest page 直接反向 tail 读取，不从头扫。

中期：写 raw frame 时生成 sidecar index：

```text
line_number, byte_offset, timestamp, direction, kind
```

查询时：

- 根据 kind/direction 过滤 index。
- 根据 offset seek 到具体行。
- content 大时只读 preview。

长期：如果 raw frame 成为核心功能，考虑 SQLite / sled / redb 存储事件索引。

#### 验证

- 10MB / 100MB raw jsonl
- raw 面板打开耗时
- 搜索耗时
- 后端 CPU 和磁盘读取量

### 4.3 task/workflow summary 存在 N+1 文件系统遍历

入口：

- `src-tauri/src/view_models.rs:458-489`
- `src-tauri/src/view_models.rs:517-538`
- `src-tauri/src/view_models.rs:540-553`
- `src-tauri/src/view_models.rs:555-581`

计数：

- `src-tauri/src/view_models.rs:1705-1749`

#### 问题

`task_list_vm` 对每个 task 构建 row：

```rust
let (artifact_count, attachment_count) = count_task_outputs(app, &summary.task.id)?;
```

`count_task_outputs`：

- 遍历 runs
- 每个 run 遍历 rounds
- 每个 round 调 `count_round_outputs`

`count_round_outputs`：

- 遍历 nodes
- 每个 node 遍历 attempts
- 每个 attempt 读 artifact list / attachment list

#### 影响

task 数、run 数、round 数增长后，任务列表会越来越慢。再叠加前端 10s refresh，问题会放大。

#### 建议

1. **运行时维护 summary index**
   - task summary 中持久化：
     - artifact_count
     - attachment_count
     - latest_run
     - latest_round
     - status
     - resumable_run_id

2. **列表页不要做深层统计**
   - 列表只读 summary。
   - detail 页再 lazy load 深层资产数量。

3. **round graph 中 artifact/attachment count 批量读取**
   - 目前每个 node/attempt 单独 list。
   - 可在 round 层一次扫描输出目录，构建 map。

#### 验证

样本：

- 100 tasks
- 每 task 20 runs
- 每 run 5 rounds
- 每 round 20 nodes

测 command：

- `get_task_list`
- `get_workflow`
- `get_round_detail`

### 4.4 worker 执行链路内存风险和 timeout 缺失

位置：

- `src/worker/mod.rs:43-141`
- 关键：`src/worker/mod.rs:87-103`

当前：

```rust
let output = shell_command(command.run.as_str())
    .current_dir(cwd.as_std_path())
    .output()?;
```

#### 问题

1. `Command::output()` 会把 stdout/stderr 完整读入内存。
2. `timeout_sec` 字段定义了，但执行时没有使用。
3. 长输出命令会造成内存峰值。
4. 卡住的命令不会被超时终止。

#### 建议

改为：

- `spawn`
- stdout/stderr pipe
- 独立线程或 async task 流式写文件
- 监控 timeout
- timeout 后 kill child
- 写入 `timed_out: true`

伪结构：

```rust
let mut child = command
  .stdout(Stdio::piped())
  .stderr(Stdio::piped())
  .spawn()?;

stream_to_file(child.stdout.take(), stdout_path);
stream_to_file(child.stderr.take(), stderr_path);

wait_with_timeout(&mut child, timeout);
```

#### 验证

命令：

- 生成 100MB stdout
- 生成 100MB stderr
- sleep 超过 timeout

观察：

- RSS 是否稳定
- timeout 是否生效
- stdout/stderr 文件是否完整写入

### 4.5 ACP permission wait 阻塞 runtime loop

位置：

- `src/acp/client.rs:396-419`
- `src/acp/permission.rs:79-91`
- channel：`src/acp/client.rs:131-144`

当前 permission 处理：

```rust
let response = wait_for_permission_response(&self.paths.attempt_dir, &request_id)?;
```

`wait_for_permission_response`：

```rust
loop {
    if path.exists() {
        // ...
        return Ok(response);
    }
    thread::sleep(Duration::from_millis(200));
}
```

#### 问题

ACP runtime 在处理 permission request 时同步等待文件出现。等待期间，如果 adapter 继续输出 event，stdout reader 线程会继续往 `sync_channel(1024)` 发；如果主 loop 不 drain，可能阻塞。

#### 建议

1. permission wait 期间继续 drain inbound。
2. 用 channel / condition variable 替代文件 polling。
3. 至少增加 timeout/cancel 检查。
4. 如果保留文件机制，sleep polling 可接受但需要不阻塞主事件处理。

#### 验证

构造场景：

- permission pending 时 adapter 持续输出 session/update。
- 检查是否卡住、channel 是否堆积、UI 是否继续更新。

### 4.6 ACP raw/event 写入存在同步频繁打开文件成本

位置：

- `src/acp/events.rs:99-129`
- `src/storage/mod.rs:310-319`
- 调用：`src/acp/client.rs:141`、`src/acp/client.rs:390`、`src/acp/client.rs:422`

当前每 append 一条 JSONL：

```rust
let mut file = OpenOptions::new()
    .create(true)
    .append(true)
    .open(path.as_std_path())?;
serde_json::to_writer(&mut file, value)?;
file.write_all(b"\n")?;
```

#### 问题

每个 frame 都 open/append/close。ACP streaming 高频时，文件系统开销明显。

#### 建议

1. ACP runtime 内部持有 `BufWriter<File>`。
2. 批量 flush，例如每 N 条或关键事件 flush。
3. raw/event/diagnostic 分别持有 writer。
4. 确保 shutdown 时 flush。

#### 权衡

- 每条都落盘更安全。
- buffer 后崩溃可能丢最后几条。
- 可以关键状态立即 flush，普通 delta 批量 flush。

### 4.7 ACP 文本拼接潜在 O(n²)

位置：

- `src/acp/client.rs:377-386`
- `src/acp/client.rs:469-479`

当前：

```rust
if target.chars().count() >= max_chars {
    return;
}
let remaining = max_chars - target.chars().count();
```

#### 问题

每个 delta 都对已有 target 做 `chars().count()`，随着文本增长，累计成本上升。

#### 建议

维护计数状态：

```rust
struct BoundedText {
    value: String,
    chars: usize,
    truncated: bool,
}
```

追加时只计算新 content 的 char count。

#### 优先级

中等。相比 JSONL 扫描和 raw frame，这不是第一瓶颈，但实现简单。

### 4.8 Tauri command 同步 I/O

位置：

- `src-tauri/src/commands.rs:48-214`
- `src-tauri/src/commands.rs:193-214`

当前很多 command 是同步函数：

```rust

pub fn get_workflow(...)
pub fn get_round_detail(...)
pub fn get_log_page(...)
pub fn get_acp_session(...)
```

部分重活已经 `spawn_blocking`：

- `send_acp_prompt`: `src-tauri/src/commands.rs:217-257`
- `get_acp_raw_frames`: `src-tauri/src/commands.rs:330-360`

#### 问题

同步 command 内部做文件遍历/JSONL 扫描，可能占用 Tauri IPC handling 线程，导致 UI 响应不稳。

#### 建议

1. 重 I/O command 改 async + `spawn_blocking`。
2. 更重要的是减少内部 I/O，而不是只移动线程。
3. command payload 保持小，raw 大内容不要默认返回。

## 8. 推荐最终目标架构

### 8.1 ACP 数据流

当前：

```text
ACP adapter
  -> append JSONL
  -> frontend polling
  -> backend full scan JSONL
  -> Tauri IPC full VM
  -> frontend merge/render
```

建议：

```text
ACP adapter
  -> append JSONL + update in-memory/index metadata
  -> Tauri event incremental push
  -> frontend append event window
  -> history/raw 按需分页读取
```

关键原则：

- active session 走增量。
- 历史查询走 cursor。
- raw 大内容按需。
- diagnostics/count 不阻塞主 chat 更新。

### 8.2 列表页数据

当前：

```text
task list render
  -> scan task
  -> scan runs
  -> scan rounds
  -> scan nodes
  -> scan attempts
  -> count artifacts/attachments
```

建议：

```text
runtime writes summary index
  -> task list reads summary only
  -> detail page lazy loads deep data
```

### 8.3 raw frame 数据

当前：

```text
query raw page
  -> count full file
  -> collect full file
  -> frontend parse all rows
```

建议：

```text
write raw frame
  -> store offset/kind/direction index

query raw page
  -> read index
  -> seek selected offsets
  -> return preview

expand one row
  -> fetch full content
  -> pretty print one row
```

## 9. 最终建议排序

如果只做 5 件事，建议按这个顺序：

1. **raw frame 前端懒解析**
2. **ACP session 后端增量读取 / 去掉每次全量 count**
3. **worker stdout/stderr 流式写入 + timeout**
4. **task/workflow summary index**
5. **ACP scroll rAF 节流 + timeline 构建去重**

这 5 项覆盖了当前最主要的响应、CPU、内存和 I/O 风险。前端移除 `react-virtuoso` 本身不是最大问题；真正需要优先解决的是 raw frame 和 ACP 后端轮询扫描。
